# pov.et ‚Äî Roadmap

> Presentation matters more than infrastructure.

The order of work matters. We build the **experience** first and the
**plumbing** behind it. If we invert this order, we waste effort plumbing for
an experience that hasn't proven it deserves to exist.

---

## Build order (do not invert)

1. **Beautiful static prototype** ‚Äî feed, post, about. Mock data only.
2. **Perfect feed UX** ‚Äî pacing, spacing, motion, image breathing.
3. **Perfect mobile experience** ‚Äî this is mobile-first, not mobile-second.
4. **Local JSON data layer** ‚Äî typed shape that matches the future Postgres
   schema.
5. **Database** ‚Äî Postgres + Prisma. Swap JSON loader for DB queries.
6. **Telegram sync** ‚Äî GramJS worker, manual trigger first, then cron.
7. **Image pipeline** ‚Äî R2 (or Supabase Storage) + variants + blurhash.
8. **Automation & ops** ‚Äî cron, alerting, retries, idempotency.
9. **Editorial tooling** ‚Äî minimal admin for `featured`, `status`, tagging.

**Do not start with backend complexity.** Even static / mock data is
acceptable for the first several iterations. The aesthetic browsing experience
**is** the real MVP.

- Lightbox with soft fade transitions, keyboard nav (‚Üê/‚Üí/Esc), and swipe
- Subtle motion via `FadeIn` (respects `prefers-reduced-motion`)
- Quiet header + footer; SEO-friendly metadata per route

## Phase 4 ‚Äî Telegram sync ‚úÖ (code-complete; awaiting credentials)

### What landed

- `lib/telegram/parser.ts` ‚Äî shared caption parser (location, contributor,
  body) used by both the bootstrap script and the live sync. Single source
  of truth for caption conventions.
- `lib/telegram/groups.ts` ‚Äî **the** correctness piece. Collapses N Telegram
  messages sharing a `groupedId` back into one logical post, with stable
  ordering, lead-caption resolution, earliest publish date, max view count,
  and the first non-empty reactions set. 12 unit tests in
  `tests/telegram-groups.test.ts`, all green.
- `lib/telegram/client.ts` ‚Äî GramJS client factory + env config reader.
  Silent logger so sync runs stay readable. Configured to never leak into
  the Next.js bundle (`serverExternalPackages`).
- `lib/telegram/media.ts` ‚Äî per-image pipeline: mild JPEG normalization,
  dimensions, blurhash, decoded blurDataURL PNG, dominant color. Honors
  the PRODUCT.md image philosophy ‚Äî no aggressive resizing, texture
  preserved.
- `lib/storage.ts` ‚Äî storage abstraction. Phase 4 implementation writes
  to `public/media/` with content-hashed keys. Phase 5 swaps the body for
  Cloudflare R2; nothing else changes.
- `lib/telegram/sync.ts` ‚Äî the orchestrator. Connects, finds the highest
  `telegramMessageId` already stored, fetches new messages, groups,
  downloads, processes, and upserts inside a per-post transaction.
  Idempotent. Reports a structured summary.
- `scripts/telegram-login.ts` ‚Äî one-time interactive auth. Prompts for
  phone, code, and 2FA password (masked). Prints a session string.
- `scripts/telegram-sync.ts` ‚Äî sync entry point. Supports `--limit N` and
  `--full` for backfills.
- npm scripts: `telegram:login`, `telegram:sync`, `test`.
- `.env.example` documents the full credential flow.
- `public/media/` added to `.gitignore`.

### Still pending

- The team must obtain `TELEGRAM_API_ID` + `TELEGRAM_API_HASH` from
  `my.telegram.org` and run `pnpm run telegram:login` once.
- First end-to-end run against the real channel (`pnpm run telegram:sync
  -- --full`).
- Scheduled invocation (cron / Vercel Cron / GH Action) ‚Äî deliberately
  punted until manual runs are stable.

### Done when

- A fresh database, after one full sync, mirrors the public channel. ‚è≥
- Re-running sync is a no-op. ‚úÖ (architecturally guaranteed by the
  upsert-on-telegramMessageId pattern; awaits real-data verification)
- New posts on the channel appear on the website within ~1 minute of the
  next sync run. ‚è≥ (depends on schedule)

## Phase 5 ‚Äî Image pipeline ‚úÖ (code-complete; awaiting R2 credentials)

### What landed

- `lib/storage.ts` now exposes a `StorageBackend` interface with two
  implementations:
  - `LocalBackend` ‚Äî current behavior (`public/media/` + `/media/<key>`
    URLs). Used in dev and as the failsafe when R2 isn't configured.
  - `R2Backend` ‚Äî Cloudflare R2 over the S3 API (`@aws-sdk/client-s3`).
    Idempotent uploads via `HeadObject`. Objects written with
    `Cache-Control: public, max-age=31536000, immutable` because keys
    are `sha1(fingerprint, bytes)` and so are content-addressed.
- Backend selection is driven by env at runtime ‚Äî if `R2_ACCOUNT_ID`,
  `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET`, and
  `R2_PUBLIC_URL` are all set, R2 is used. Otherwise we fall back to
  local. 5 unit tests cover the selection logic, all green.
- The first line of `pnpm run telegram:sync` now identifies the active
  backend (`[sync] storage: Cloudflare R2 ‚Ä¶` or `[sync] storage: local
  filesystem (public/media/)`), so it's never ambiguous which one is
  serving the run.
- `next.config.mjs` derives `next/image`'s `remotePatterns` from
  `R2_PUBLIC_URL` (and blanket-allows `*.r2.dev`), so neither dev nor
  prod needs any further image-config wiring.
- `.env.example` and `README.md` carry the full Cloudflare setup
  walkthrough (bucket creation, public access, API token, custom domain).
- Content-hashed keys + blurhash + dominant color were already cached on
  `Media` from Phase 4; nothing new needed there.

### Still pending

- Operator-side: create the R2 bucket, mint an API token, set
  `R2_PUBLIC_URL`, fill the five vars in `.env`, re-run the sync. The
  re-run rewrites every Media row's URL atomically (idempotent upserts).
- (Optional, later) image variants (`full`, `feed`, `thumb`). Phase 4's
  pipeline currently produces one full-size JPEG per photograph. Variants
  are a nice-to-have for bandwidth on mobile but not required ‚Äî `next/image`
  derives responsive sizes on its own from the source.

### Done when

- The Telegram sync uploads to R2 (verified by the storage log line). ‚è≥
- Every photograph URL on the live site resolves to the R2 public base. ‚è≥
- Re-running sync over already-uploaded photos is a no-op (the
  `HeadObject` short-circuit). ‚úÖ architecturally guaranteed.
- Texture and grain still feel like phone photos, not stock images. ‚úÖ
  the image pipeline doesn't resize aggressively.

## Out of scope (V1)

- accounts
- comments
- notifications
- a follow system
- on-site upload
- a social graph
- public REST / GraphQL API
- a mobile app

Telegram handles submissions. The website is **read-only** for V1.

## Definition of done ‚Äî the project

> When a user opens pov.et and silently scrolls for minutes, the work is
> working.

When that is true on mobile and on desktop, when locations feel like places,
when contributors feel like authors, and when an everyday photo from a street
in Addis is given the dignity of a page ‚Äî V1 is done.
