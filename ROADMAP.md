# pov.et — Roadmap

> Presentation matters more than infrastructure.

The order of work matters. We build the **experience** first and the
**plumbing** behind it. If we invert this order, we waste effort plumbing for
an experience that hasn't proven it deserves to exist.

---

## Build order (do not invert)

1. **Beautiful static prototype** — feed, post, about. Mock data only.
2. **Perfect feed UX** — pacing, spacing, motion, image breathing.
3. **Perfect mobile experience** — this is mobile-first, not mobile-second.
4. **Local JSON data layer** — typed shape that matches the future Postgres
   schema.
5. **Database** — Postgres + Prisma. Swap JSON loader for DB queries.
6. **Telegram sync** — GramJS worker, manual trigger first, then cron.
7. **Image pipeline** — R2 (or Supabase Storage) + variants + blurhash.
8. **Automation & ops** — cron, alerting, retries, idempotency.
9. **Editorial tooling** — minimal admin for `featured`, `status`, tagging.

**Do not start with backend complexity.** Even static / mock data is
acceptable for the first several iterations. The aesthetic browsing experience
**is** the real MVP.

---

## Where we are

**Phases 1–5 live in production.** The archive runs on Vercel against
Neon Postgres, with images served from Cloudflare R2 and the channel
syncing every 20 minutes via GitHub Actions.

**Next up: Phase 6 — Editorial tooling.** The owner needs a quiet, locked
admin route to toggle `featured` / `status` and assign mood tags without
touching the database directly.

## Phase 1 — Static prototype ✅

The goal of this phase: when someone opens the site for the first time, they
**stop scrolling other things**.

### What landed

- Next.js 15 (App Router) + TypeScript + Tailwind + Framer Motion
- Color tokens (`paper`, `ink`, `sky`, `moss`, `mist`, `soot` + a derived
  slate scale), display type (Fraunces variable) + body type (Inter)

  preview at `t.me/s/pov_et`, downloads photographs to `public/mock/` with
  content-hashed keys, generates blurhashes + tiny PNG data URLs, and
  writes a typed `lib/mock-data.json` mirroring the future Prisma schema
- Routes:
  - `/` — immersive feed (ISR every 5 minutes)
  - `/post/[slug]` — single post, fullscreen viewer with keyboard +
    swipe navigation
  - `/location/[slug]` — emotional map per place (statically generated)
  - `/photographer/[username]` — contributor archive (dynamic)
  - `/about` — drafted manifesto
  - `/submit` — quiet interstitial linking to `@povetbot`
  - `/not-found` — custom 404 in the same voice
- Mobile-first CSS-column masonry feed
- `next/image` with blurhash placeholders embedded as base64 PNG data URLs
- Lightbox with soft fade transitions, keyboard nav (←/→/Esc), and swipe
- Subtle motion via `FadeIn` (respects `prefers-reduced-motion`)
- Quiet header + footer; SEO-friendly metadata per route

### Verification

- `pnpm run build` — green, 24 pages prerendered, ~150 kB First Load JS
  on photograph pages
- No linter errors
- Visual smoke test on desktop confirmed feed, post, lightbox, about,
  location all render correctly

## Phase 4 — Telegram sync ✅

### What landed

- `lib/telegram/parser.ts` — shared caption parser (location, contributor,
  body) used by both the bootstrap script and the live sync. Single source
  of truth for caption conventions.
- `lib/telegram/groups.ts` — **the** correctness piece. Collapses N Telegram
  messages sharing a `groupedId` back into one logical post, with stable
  ordering, lead-caption resolution, earliest publish date, max view count,
  and the first non-empty reactions set. 12 unit tests in
  `tests/telegram-groups.test.ts`, all green.
- `lib/telegram/client.ts` — GramJS client factory + env config reader.
  Silent logger so sync runs stay readable. Configured to never leak into
  the Next.js bundle (`serverExternalPackages`).
- `lib/telegram/media.ts` — per-image pipeline: mild JPEG normalization,
  dimensions, blurhash, decoded blurDataURL PNG, dominant color. Honors
  the PRODUCT.md image philosophy — no aggressive resizing, texture
  preserved.
- `lib/storage.ts` — storage abstraction. Phase 4 implementation writes
  to `public/media/` with content-hashed keys. Phase 5 swaps the body for
  Cloudflare R2; nothing else changes.
- `lib/telegram/sync.ts` — the orchestrator. Connects, finds the highest
  `telegramMessageId` already stored, fetches new messages, groups,
  downloads, processes, and upserts inside a per-post transaction.
  Idempotent. Reports a structured summary.
- `scripts/telegram-login.ts` — one-time interactive auth. Prompts for
  phone, code, and 2FA password (masked). Prints a session string.
- `scripts/telegram-sync.ts` — sync entry point. Supports `--limit N` and
  `--full` for backfills.
- npm scripts: `telegram:login`, `telegram:sync`, `test`.
- `.env.example` documents the full credential flow.
- `public/media/` added to `.gitignore`.

### Production

- `TELEGRAM_API_ID` + `TELEGRAM_API_HASH` issued from `my.telegram.org`,
  one-time `pnpm run telegram:login` performed, session string stored as
  a repo secret + Vercel env var.
- First full backfill run against the real channel completed.
- Scheduled invocation: GitHub Actions cron (`.github/workflows/telegram-sync.yml`)
  runs every 20 minutes incrementally.

### Done when

- A fresh database, after one full sync, mirrors the public channel. ✅
- Re-running sync is a no-op. ✅ (architecturally guaranteed by the
  upsert-on-telegramMessageId pattern; verified against real data.)
- New posts on the channel appear on the website within ~20 minutes of
  the next scheduled sync run. ✅

## Phase 5 — Image pipeline ✅

### What landed

- `lib/storage.ts` now exposes a `StorageBackend` interface with two
  implementations:
  - `LocalBackend` — current behavior (`public/media/` + `/media/<key>`
    URLs). Used in dev and as the failsafe when R2 isn't configured.
  - `R2Backend` — Cloudflare R2 over the S3 API (`@aws-sdk/client-s3`).
    Idempotent uploads via `HeadObject`. Objects written with
    `Cache-Control: public, max-age=31536000, immutable` because keys
    are `sha1(fingerprint, bytes)` and so are content-addressed.
- Backend selection is driven by env at runtime — if `R2_ACCOUNT_ID`,
  `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET`, and
  `R2_PUBLIC_URL` are all set, R2 is used. Otherwise we fall back to
  local. 5 unit tests cover the selection logic, all green.
- The first line of `pnpm run telegram:sync` now identifies the active
  backend (`[sync] storage: Cloudflare R2 …` or `[sync] storage: local
  filesystem (public/media/)`), so it's never ambiguous which one is
  serving the run.
- `next.config.mjs` derives `next/image`'s `remotePatterns` from
  `R2_PUBLIC_URL` (and blanket-allows `*.r2.dev`), so neither dev nor
  prod needs any further image-config wiring.
- `.env.example` and `README.md` carry the full Cloudflare setup
  walkthrough (bucket creation, public access, API token, custom domain).
- Content-hashed keys + blurhash + dominant color were already cached on
  `Media` from Phase 4; nothing new needed there.

### Production

- R2 bucket provisioned, public URL configured, API token minted. The
  five `R2_*` env vars are set on Vercel and in the sync workflow.
- Sync re-run rewrote every `Media.imageUrl` to the R2 public base
  atomically (idempotent upserts via `HeadObject`).
- (Optional, later) image variants (`full`, `feed`, `thumb`). The current
  pipeline produces one full-size JPEG per photograph. Variants are a
  nice-to-have for bandwidth on mobile but not required — `next/image`
  derives responsive sizes on its own from the source.

### Done when

- The Telegram sync uploads to R2 (verified by the storage log line). ✅
- Every photograph URL on the live site resolves to the R2 public base. ✅
- Re-running sync over already-uploaded photos is a no-op (the
  `HeadObject` short-circuit). ✅
- Texture and grain still feel like phone photos, not stock images. ✅
  the image pipeline doesn't resize aggressively.

## Out of scope (V1)

- accounts
- comments
- notifications
- a follow system
- on-site upload
- a social graph
- public REST / GraphQL API
- a mobile app

Telegram handles submissions. The website is **read-only** for V1.

---

## Definition of done — the project

> When a user opens pov.et and silently scrolls for minutes, the work is
> working.

When that is true on mobile and on desktop, when locations feel like places,
when contributors feel like authors, and when an everyday photo from a street
in Addis is given the dignity of a page — V1 is done.
